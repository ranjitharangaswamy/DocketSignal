01 · What I built

A public discourse research pipeline and static showcase

I built DocketSignal, a small research pipeline and static dashboard for analyzing public Reddit discourse about legal AI. The tool collects or loads posts and comments about legal AI, cleans the text, assigns early qualitative themes, adds sentiment and rhetorical-frame signals, and turns the results into CSV files, researcher memos, a chart, and a public-facing dashboard. It is designed for legal-tech researchers, UX researchers, and product collaborators who want to understand what practitioners publicly argue about: hallucinated citations, trust, billing pressure, adoption resistance, governance, workflow gains, and tool choice.

The final artifact has two layers. The Python layer creates structured qualitative artifacts from public text. The showcase layer explains those artifacts through a portfolio-style interface. I also built a design thinking page that documents why the evidence panel does not use one generic card layout for every theme. Instead, it treats each theme as a different discourse type: accuracy appears as citation-like warning blocks, adoption resistance as a sticky-note culture map, ethics as policy memo language, workflow gains as handoff steps, role anxiety as persona-oriented evidence, vendor comparison as a split product/thread view, eDiscovery as affect plus evidence, and cost as invoice or ledger logic.

02 · Decisions

Reddit-only data, transparent coding, and theme-specific visual forms

I chose a Cursor + Python project because the core work is data collection, cleaning, and analysis rather than a production web app. I kept Reddit as the live data source because it has public practitioner-facing discussion and can be accessed through an official API with PRAW. I dropped broader web and LinkedIn scraping from the final scope because those sources raise more terms-of-service and provenance issues, and they would have made the project less reliable for a class deliverable.

The biggest design decision was to separate ranking from interpretation. A frequency chart is useful for answering where discourse concentrates, but it cannot express what different arguments feel like to read. The design thinking page makes this claim directly: one chart cannot carry eight different arguments. I chose to match metaphor to discourse type rather than theme label alone. That is why the dashboard keeps a consistent legal research palette, provenance fields, and source links while allowing the evidence layouts to vary.

03 · What I would do differently

Add validation loops and test the metaphor system

I would improve the qualitative coding layer by adding a human validation loop directly into the tool. Right now the themes are transparent and inspectable, but the researcher still has to validate them outside the interface. A stronger version would let the researcher accept, reject, rename, or merge codes in the dashboard and then export a revised codebook.

On the design side, I would user-test the metaphor system. The design thinking page argues that eight themes need eight visual languages, but that is still a design hypothesis. I would test whether legal researchers actually remember theme definitions better when evidence panels have distinct forms, or whether some users prefer a uniform card system for speed. I would also add a simple mode toggle: one mode for portfolio storytelling and one mode for dense research review.

04 · Competency demonstration

Computational fluency, data ethics, qualitative judgment, and design rationale

This project demonstrates computational fluency, data ethics, qualitative research judgment, and communication design. The computational work appears in src/pipeline.py, src/collect_reddit.py, and src/qualitative.py, which collect, clean, code, summarize, and export the corpus. The ethical scope is visible in the Reddit-only collection strategy, .env.template, provenance notes, and repeated disclaimer that the tool supports analysis rather than replacing interpretation.

The design thinking page adds a more specific interface-design claim: visualization should respect the substance of each code, not only its count. The project demonstrates that claim through the theme evidence system in theme-evidence.js, the design documentation in design.html, and the shared styling in styles.css. The interface keeps consistent provenance, subreddit, score, and source-link details while changing the visual metaphor for different discourse patterns.
